How to use the Token Editor
✕
Goal: Clean up place-name tags so the Itinerary Builder only shows correct product chips. Fixes are saved to token_corrections.json and re-applied automatically every time prepare_data.py runs — fix each issue once, it stays fixed.
Step 1 — Filter to flagged tokens
Use Status → Needs review to see only tokens the tagger has flagged. There are 6 flag types, shown in priority order:
Not a place?Possessive form like Perth’s. Always remove — click the red button.
Should splitTwo places joined by & (e.g. Kakadu & Katherine). Split them into two separate products.
Extra wordAction verb or generic noun stuck to a place name — e.g. Arrive Broome or Cable Beach transfer. Rename to the place only.
Duplicate?Near-duplicate of a more popular token (e.g. uluru vs Uluru). Merge into the canonical version to combine popularity counts.
ALL CAPS / lowercaseToken extracted in wrong case. Click the recommended fix to normalise to title case.
Step 2 — Review each token
Click any row to open the detail panel on the right. You will see the flag description, location examples where it was extracted from, and the suggested fix.
Recommended fix button — green = rename, red = remove. One click applies it.
Not right? Edit manually — opens fields to type a custom name, merge into an existing product, or define custom split parts.
Keep as-is — Approve the token if it is actually correct despite the flag.
Not a place — Remove it from the product list entirely.
Step 3 — Export and apply
1
Click Preview & sync to database at the top of this page — review the full change list.
2
Click Download token_corrections.json.
3
Move the file into the Build and explore folder (replaces the old one).
4
Run python prepare_data.py then reload the Itinerary Builder. Fixed chips appear immediately.
Important: Changes live only in this browser session. Download before closing the tab or you will lose unsaved work.
Tips
• Filter by State to work on one region at a time.
• The Day rows column shows how many itinerary day-rows this token appears in — higher means more impact when fixed.
• Source: B = Board only, C = Competitor only, BC = both. Board tokens are high-value.
• On the next pipeline run, all past corrections in token_corrections.json are re-applied automatically — no re-editing needed. Just review new flags.
